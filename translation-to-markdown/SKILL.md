# 从 URL 到译文 Markdown

当用户提供**源文章 URL**，并希望获得可保存为博客文章的**完整中文译本**时使用本技能。

细则与长提示词集中在 [`references/prompt.md`](references/prompt.md)，本文件不重复粘贴，仅约定**独有规则**与**必读指针**。

## 产出内容

生成一篇可直接用于 Astro 文章页面的 **Markdown 全文**，包含：

1. YAML frontmatter（元数据）
2. 开篇版权与免责声明（blockquote）
3. 简体中文译文正文（**正文抽取：保留/排除哪些模块**见 `prompt.md`「正文抽取」）
4. `## 译者总结`（**撰写要求、禁止项、可选子结构与执行用提示词全文**见 `prompt.md`「译者总结 · 执行用提示词」）
5. 术语表**仅**允许出现在 `## 译者总结` 内；条数与格式以 `prompt.md` 为准，不得单独成章

不要在 Markdown 正文中输出环境说明、保存路径建议、git 命令、HTML 注释或配图清单类说明。

## 输入约定

默认输入为：

- **一个**源文章 URL

除非用户明确要求覆盖，否则不要向用户索要 slug、tags、date、图片目录或文件名。

## 核心流程

1. **抓取与抽取**：按 [`references/prompt.md`](references/prompt.md)「正文抽取」保留主体、排除导航/广告等；不可靠则**不要编造**，说明原因并请用户粘贴全文或可访问镜像。
2. **翻译正文**：自然、流畅的**简体中文**；保留论证顺序；代码块保留 fenced 语言标记（首行如 `ts`、`bash`、`text` 等）；命令、API、路径、版本号与专名按原文需要可保留英文。
3. **标题**：遵守下文「标题规则」（译文主体不得乱加标题；`## 译者总结` 及其内部结构以 `prompt.md` 为准）。
4. **译者总结**：正文之后必有 `## 译者总结`；具体写法以 `prompt.md` 同节内 **text 代码围栏中的全文**为准，整段作为 system 或对话前置说明。
5. **配图**：遵守下文「配图规则」。

## Frontmatter 规则

使用如下结构：

```yaml
---
title: "【译】中文标题"
date: YYYY-MM-DD HH:mm:00+08:00
draft: true
slug: "derived-slug"
categories: [ "translation" ]
tags: [ "tag1", "tag2" ]
description: "一两句中文摘要"
canonicalURL: "https://original-url"
---
```

字段说明：

- `title`
  - 使用双引号
  - 以 `【译】` 开头
  - 使用自然中文
  - 贴近原文语气
  - 产品名、技术专名在需要时可保留英文

- `date`
  - 默认表示**译文落稿 / 发布日期**：使用当前 `Asia/Shanghai` 时间
  - 格式为 `YYYY-MM-DD HH:mm:00+08:00`
  - 若无法可靠取得当前时间，应说明情况，不要瞎猜
  - 若用户明确要求使用原文发布日或其他日期，按其说明填写

- `draft`
  - 默认一律为 `true`，除非用户明确要求否则

- `slug`
  - 必须取 URL 路径里最后一段有意义的片段
  - 不能自行推导
  - 去掉 `.html`、`.htm` 等无意义后缀

- `categories`
  - 固定为 `[ "translation" ]`

- `tags`
  - 1～3 个**英文**标签
  - 优先使用与文章主题直接相关的具体技术词
  - 避免空洞标签

- `description`
  - 1～2 句中文
  - 克制、写实

- `canonicalURL`
  - 必须与用户输入的原文 URL **完全一致**

## 译文忠实度

- 以**逐段忠实翻译**为优先。
- 不要主动做编辑式「润色」重写。
- 除非不调整会导致中文明显拗口或难读，否则不要仅为「更顺」而改写作者的句序与推进方式。
- 在中文仍能承载的前提下，保留作者的修辞节奏、过渡与局部强调。
- 仅在为了中文清晰或语法正确时拆分长句。

## 标题规则（译文主体）

以下约束**仅针对译文正文主体**，不包括 `prompt.md` 规定可写在 `## 译者总结` 内部的 `###` 子标题。

- 必须保留原文**标题层级结构**（有几个标题、各是什么层级，译文一一对应）。
- 标题文字译为中文。
- `Introduction`、`Conclusion`、`References` 这类通用标题**不要**留在英文。
- 标题中仅专名、产品名、协议名或固定技术术语可保留英文。
- **禁止添加**原文正文中**不存在**的任何 `h1`～`h6`（含 Markdown `#`～`######`）。禁止仅为「好读」「像博客」而新增小节。
- **禁止**凭空使用常见小节名作为新标题，例如：摘要、概述、背景、要点、结论、小结、延伸阅读等——**除非**原文在对应位置已有同级标题。
- 若原文主体仅为**连续段落**、无小节标题，译文须保持连续段落结构，不得拆成多个 `##`/`###`。
- 文章主标题由 frontmatter 的 `title` 承担；若原文正文**没有**一级标题，译文正文**不要**再用单个 `# …` 重复文章题目。若原文正文**含有**一级标题，则按原文翻译，不删不改层级。
- 不要仅为「版面好看」而合并、打乱或重排标题。

**反例（不要输出）：**

- 原文无「## 摘要」，译文却加 `## 摘要`。
- frontmatter 已有 `title`，正文再来一行 `# 原标题` 当主标题（而原文正文并无该 `#`）。

## 开篇引用块

在 frontmatter **之后**立即插入如下引用块：

```md
> 本文为学习目的的个人翻译，译文及后文「译者总结」仅供参考。
>
> 原文链接：[原文标题](https://example.com/article)。
>
> 版权归原作者或原刊登方所有。本文为非官方译本；如有不妥，请联系删除。
```

语气保持克制。不要额外加 `## 翻译说明` 一类标题。

## 配图规则

属于正文主体的配图：

- 全部保留
- 保持正文中的出现顺序
- Markdown 中写为：

```md
![说明](01.webp)
![说明](02.webp)
```

- Markdown 里**只写文件名**
- 路径**不要**以 `/` 开头
- **不要**把原文 hotlink 当作文章长期使用的图片地址

预期存放路径：

```text
public/images/{slug}/
```

编号：

- `01.webp`、`02.webp`、`03.webp`…
- 同一张图再次出现时**复用**同一文件名，不要重新编号

正文广告图、赞助图或与正文无关的装饰图不要保留。

若拿不到稳定可用的图片地址，在 **Markdown 正文之外**说明，不要虚构。

## 失败时的处理

若因付费墙、登录墙、反爬、页面结构劣质或无法定位可靠正文而导致抽取失败：

- 不要编造任何内容
- 只输出失败说明与下一步需要什么
- 请用户粘贴全文或提供可访问镜像
- 不要输出虚假 frontmatter 或半成品正文

## 质量自检清单

回应前确认：

- 正文抽取符合 [`references/prompt.md`](references/prompt.md)「正文抽取」
- 正文确实来自原文主体，而非混入了站外模块
- **标题**：除开篇引用块与 `## 译者总结`（及 `prompt.md` 允许其下的 `###` 子标题）外，译文主体里**每一个** `#`～`######` 在原文中均有对应
- 逻辑、否定、条件与数字未被扭曲
- 图片引用使用 `NN.webp`
- `## 译者总结` 符合 [`references/prompt.md`](references/prompt.md) 中执行用提示词，且未代替正文
- tags 为 1～3 个与文章直接相关的英文词
